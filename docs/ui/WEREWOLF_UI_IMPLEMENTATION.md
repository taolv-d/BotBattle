## 一、概述

### 1.1 项目背景

BotBattle 是一个业余时间开发的 toy 项目，定位为 AI 驱动的社交推理游戏引擎。当前阶段主要支持单人调试模式（上帝视角），后续扩展为多人在线参与的游戏。

### 1.2 文档目标

本文档描述狼人杀上帝视角 Web UI 的设计方案，包括：
- 系统架构
- 配置文件设计
- UI 布局与交互
- 事件推送机制
- 日志格式规范

### 1.3 使用场景

| 场景 | 描述 | 用户 |
|------|------|------|
| **单机调试** | 开发者以上帝视角观察 AI 互斗 | 开发者 |
| **复盘分析** | 查看游戏历史记录和复盘报告 | 开发者/测试者 |
| **功能测试** | 验证游戏逻辑和 AI 行为 | 开发者 |
| **玩家模式** | 真人参与游戏，Agent 提供建议 | 真人玩家 |

### 1.4 实现阶段

| 阶段 | 功能范围 | 状态 |
|------|---------|------|
| **第一阶段** | 上帝视角（单机调试） | 待实现 |
| **第二阶段** | 玩家模式（真人参与 + AI 辅助） | 预留接口 |

### 1.5 玩家模式详解（第二阶段）

**核心概念**: 真人玩家 + AI Agent 辅助

```
┌─────────────────────────────────────────────────────────────┐
│  玩家模式游戏流程                                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 玩家选择参与游戏                                         │
│     └─→ 选择玩家编号（如 3 号）                                │
│                                                             │
│  2. 游戏开始，玩家看到自己的身份                              │
│     └─→ 3 号玩家，身份：预言家                                │
│                                                             │
│  3. 轮到自己发言时，Agent 提供建议                            │
│     └─→ Agent 分析局势，建议发言内容                          │
│                                                             │
│  4. 玩家决策                                                 │
│     ├─→ 采纳建议（一键使用）                                 │
│     ├─→ 修改建议（编辑后发送）                               │
│     └─→ 忽略建议（自己输入）                                 │
│                                                             │
│  5. 发言发送到聊天窗口，其他玩家可见                          │
│     └─→ 真人发言 vs AI 发言 同样显示                          │
│                                                             │
│  6. 夜晚行动（预言家查验、女巫用药等）                        │
│     └─→ Agent 建议查验目标，玩家决定                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Agent 建议内容**:
- **发言建议**: 根据当前局势，建议发言内容
- **投票建议**: 建议投票给谁
- **技能建议**: 预言家查验谁、女巫救/毒谁、守卫守护谁
- **局势分析**: 当前形势判断、可疑玩家列表

**隐私保护**:
- 每个真人玩家只能看到自己的 Agent 建议
- 不能看到其他玩家的 Agent 建议
- 不能看到其他玩家的真实身份（除非死亡）

## 二、系统架构

### 2.1 架构决策

**核心原则**: 轻量级、简单、易维护

| 决策点 | 方案 | 理由 |
|--------|------|------|
| **后端框架** | Python `http.server`（内置库） | 无需额外依赖，适合 toy 项目 |
| **前端框架** | 原生 HTML + CSS + JavaScript | 无需构建工具，直接运行 |
| **通信方式** | SSE（服务器推送）+ HTTP POST（客户端指令） | 实时推送，双向通信分离 |
| **部署方式** | 一体部署（Python serve 静态文件） | 简化部署，单进程运行 |
| **状态管理** | 前端本地状态 | 单页面应用，无需复杂状态管理 |

**注意**: SSE 方案详见第六章，该方案仍需进一步讨论可行性。

## 三、配置文件设计

### 3.1 配置文件结构

配置文件位于 `config/system.json`，包含 LLM、Web UI、日志等配置。

**注意**: 第一阶段简化配置流程，移除配置页面，直接读取 JSON 配置文件启动游戏。

```json
{
  "llm": {
    "provider": "dashscope",
    "api_key": "sk-your-actual-api-key",
    "model": "qwen-plus",
    "base_url": "https://dashscope.aliyuncs.com/compatible-mode/v1",
    "temperature": 0.7,
    "timeout": 180,
    "retry_count": 5,
    "retry_delay": 3
  },

  "web_ui": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8000,
    "auto_open_browser": false,
    "poll_interval": 5,
    "show_inner_thoughts": true
  },

  "logger": {
    "console_output": true,
    "file_output": true,
    "json_event_log": true,
    "max_memory_entries": 1000,
    "max_file_size_mb": 10,
    "backup_count": 10
  }
}
```

### 3.2 配置项说明

#### 3.2.1 LLM 配置 (`llm`)

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `provider` | string | - | LLM 提供商（dashscope/deepseek/kimi/ollama） |
| `api_key` | string | - | API Key |
| `model` | string | - | 模型名称 |
| `base_url` | string | - | API 端点 URL |
| `temperature` | float | 0.7 | 创造性（0-1） |
| `timeout` | int | 180 | 请求超时（秒） |
| `retry_count` | int | 5 | 重试次数 |
| `retry_delay` | int | 3 | 重试间隔（秒） |

#### 3.2.2 Web UI 配置 (`web_ui`)

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `enabled` | boolean | true | 是否启用 Web UI |
| `host` | string | "0.0.0.0" | 监听地址（0.0.0.0 表示允许外部访问） |
| `port` | int | 8000 | 监听端口 |
| `auto_open_browser` | boolean | false | 是否自动打开浏览器（无显示器环境设为 false） |
| `show_inner_thoughts` | boolean | true | 是否显示内心独白（上帝视角） |
| `default_theme` | string | "dark" | 默认主题（dark/light） |
| ~~`poll_interval`~~ | ~~int~~ | ~~5~~ | **已废弃**，改用 SSE 实时推送 |
| ~~`auto_open_browser`~~ | ~~boolean~~ | ~~false~~ | **第二阶段预留** |

**注意**: 
- `show_inner_thoughts` 仅控制上帝视角的默认显示状态，UI 上提供开关可动态切换
- `poll_interval` 已废弃，改用 SSE 实时推送方案（见第六章）

#### 3.2.3 日志配置 (`logger`)

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `console_output` | boolean | true | 是否输出到控制台 |
| `file_output` | boolean | true | 是否输出到文件 |
| `json_event_log` | boolean | true | 是否生成 JSON 事件日志（供 UI 调试） |
| `max_memory_entries` | int | 1000 | 内存缓存最大条目数 |
| `max_file_size_mb` | int | 10 | 单个日志文件最大大小（MB） |
| `backup_count` | int | 10 | 保留的历史日志文件数 |

## 六、事件推送机制

### 6.1 通信方案设计

**第一阶段采用方案**: SSE (Server-Sent Events) + HTTP POST

| 通信方向 | 技术方案 | 用途 | 状态 |
|---------|---------|------|------|
| **服务器→客户端** | SSE | 游戏事件、玩家发言、系统消息 | 第一阶段 |
| **客户端→服务器** | HTTP POST | 玩家发言、投票、技能使用 | 第一阶段 |

**注意**: HTTP 轮询方案（5 秒间隔）已废弃，改用 SSE 实时推送。但 SSE 方案的实现细节仍需进一步讨论。

### 6.2 SSE 推送方案（待讨论）

**SSE (Server-Sent Events)** 是一种单向通信协议，允许服务器向浏览器推送实时事件。

#### 6.2.1 工作原理

```
┌──────────────┐                              ┌──────────────┐
│   浏览器      │                              │   服务器      │
└──────┬───────┘                              └──────┬───────┘
       │                                             │
       │  1. 建立 SSE 连接                             │
       │  GET /api/events/stream                     │
       │ ──────────────────────────────────────────→ │
       │                                             │
       │  2. 保持连接，推送事件                        │
       │  ←─────────────────────────────────────────  │
       │     data: {"type": "day_start", ...}        │
       │     data: {"type": "speech", ...}           │
       │                                             │
       │  3. 玩家输入发言，点击发送                    │
       │     POST /api/player/speech                │
       │ ──────────────────────────────────────────→ │
       │     {"player_id": 1, "content": "..."}      │
       │                                             │
       │  4. 服务器处理发言，通过 SSE 推送给所有客户端   │
       │  ←─────────────────────────────────────────  │
       │     data: {"type": "speech",                │
       │            "player_id": 1,                  │
       │            "content": "..."}                │
       │                                             │
```

#### 6.2.2 服务端实现示例

```python
# web/server.py

def do_GET(self):
    parsed = urlparse(self.path)
    
    if parsed.path == '/api/events/stream':
        self._handle_sse_stream()
    elif parsed.path.startswith('/api/'):
        self._handle_api_get(parsed)
    else:
        super().do_GET()

def _handle_sse_stream(self):
    """SSE 事件流"""
    self.send_response(200)
    self.send_header('Content-Type', 'text/event-stream')
    self.send_header('Cache-Control', 'no-cache')
    self.send_header('Connection', 'keep-alive')
    self.end_headers()
    
    try:
        last_event_id = 0
        while True:
            # 从 EventBus 获取新事件
            events = event_bus.get_events_since(last_event_id)
            
            for event in events:
                last_event_id = event['id']
                # SSE 格式：data: {...}\n\n
                self.wfile.write(
                    f"data: {json.dumps(event)}\n\n".encode()
                )
                self.wfile.flush()
            
            # 没有新事件时，发送心跳保持连接
            if not events:
                self.wfile.write(b": heartbeat\n\n")
                self.wfile.flush()
            
            time.sleep(0.5)  # 短暂休眠，避免空转
            
    except (BrokenPipeError, ConnectionResetError):
        # 客户端断开连接
        pass

def do_POST(self):
    """处理客户端指令"""
    parsed = urlparse(self.path)
    
    if parsed.path == '/api/player/speech':
        self._handle_player_speech()
    elif parsed.path == '/api/player/vote':
        self._handle_player_vote()
    # ...
```

**事件总线接口**:

```python
# web/event_bus.py
class EventBus:
    """事件总线 - 解耦游戏和 Web Server"""
    
    def __init__(self):
        self._events = deque(maxlen=1000)  # 事件队列
        self._game_state = None            # 游戏状态引用
        self._game_started = False         # 游戏是否已启动
    
    def set_game_state(self, game_state):
        """设置游戏状态引用"""
        self._game_state = game_state
    
    def push(self, event: dict):
        """推送事件"""
        event["timestamp"] = datetime.now().isoformat()
        self._events.append(event)
    
    def get_events(self, since: str = None) -> list:
        """获取事件（支持 since 参数）"""
        if since is None:
            return list(self._events)
        return [e for e in self._events if e["timestamp"] > since]
    
    def get_status(self) -> dict:
        """获取游戏状态"""
        if self._game_state is None:
            return {"status": "waiting"}
        return {
            "game_id": self._game_state.game_id,
            "status": "playing" if not self._game_state.game_over else "finished",
            "day_number": self._game_state.day_number,
            "president_id": self._game_state.president_id,
            "alive_count": len(self._game_state.get_alive_players()),
            # ...
        }
    
    def wait_for_start(self):
        """等待游戏启动（阻塞）"""
        while not self._game_started:
            time.sleep(0.5)
    
    def start_game(self):
        """启动游戏"""
        self._game_started = True
```

from http.server import HTTPServer, SimpleHTTPRequestHandler
import json
from urllib.parse import urlparse, parse_qs

class WerewolfHandler(SimpleHTTPRequestHandler):
    def do_GET(self):
        parsed = urlparse(self.path)
        
        # API 路由
        if parsed.path.startswith('/api/'):
            self.handle_api(parsed)
        # 静态文件
        else:
            super().do_GET()
    
    def handle_api(self, parsed):
        # 解析路径和参数
        query = parse_qs(parsed.query)
        
        # 路由处理
        if '/status' in parsed.path:
            self.send_json(get_game_status())
        elif '/events' in parsed.path:
            since = query.get('since', [None])[0]
            self.send_json(get_events(since))
        # ...

        self.send_response(200)
        self.send_header('Content-Type', 'application/json')
        self.end_headers()
        self.wfile.write(json.dumps(data).encode())

### 9.3 前端轮询实现

```javascript
// app.js
const CONFIG = {
  pollInterval: 5000,  // 5 秒
  gameId: null,
  lastTimestamp: null
};

async function init() {
  // 从 URL 或 localStorage 获取 game_id
  CONFIG.gameId = getGameIdFromUrl();
  
  // 开始轮询
  pollEvents();
}

async function pollEvents() {
  try {
    const url = CONFIG.lastTimestamp
      ? `/api/game/${CONFIG.gameId}/events?since=${CONFIG.lastTimestamp}`
      : `/api/game/${CONFIG.gameId}/events`;
    
    const response = await fetch(url);
    const data = await response.json();
    
    // 处理新事件
    data.events.forEach(event => {
      renderEvent(event);
    });
    
    // 更新最后时间戳
    if (data.events.length > 0) {
      CONFIG.lastTimestamp = data.next_since;
    }
    
  } catch (error) {
    console.error('轮询失败:', error);
  }
  
  // 继续轮询
  setTimeout(pollEvents, CONFIG.pollInterval);
}

## 十三、实现注意事项

### 13.1 模块解耦

**核心原则**: Web Server 不直接依赖 Orchestrator

- ✅ 使用 `EventBus` 作为中间层
- ✅ `WebUI` 适配器推送事件到 EventBus
- ✅ Web Server 从 EventBus 拉取数据
- ❌ 避免 Web Server 直接访问 `orchestrator.state`

### 13.2 内心独白处理

**配置项**: `web_ui.show_inner_thoughts: true`

**实现流程**:
```
Orchestrator 调用 UI.display_inner_thought()
    ↓
WebUI.display_inner_thought() 推送事件到 EventBus
    ↓
EventBus 存储事件
    ↓
前端轮询 /api/events 获取事件
    ↓
前端渲染内心独白（斜体、浅色背景）
```

**注意**:
- 内心独白同步显示在聊天框中
- 显示在对应玩家发言之后
- 前端不提供关闭开关（由配置控制）

### 13.3 玩家面板信息

**上帝视角**:
- 显示所有玩家的真实身份
- 显示各角色的技能状态
- 狼人显示刀人信息
- 预言家显示查验信息
- 女巫显示药剂使用状态
- 守卫显示守护信息

**玩家模式（第二阶段）**:
- 只显示自己的身份
- 其他玩家只显示基本信息（存活状态、人格）
- 死亡玩家显示身份（死后公开）

### 13.4 游戏启动流程

**关键**: 用户点击"开始游戏"后才创建 Orchestrator

```
1. werewolf_main.py 启动
   ↓
2. 加载 config/system.json
   ↓
3. 创建 EventBus、LoggerService、WebServer
   ↓
4. 启动 Web Server（监听 8000 端口）
   ↓
5. 显示配置页面（config.html）
   ↓
6. 用户点击"开始游戏"
   ↓
7. POST /api/start 提交配置
   ↓
8. 创建 Orchestrator、初始化游戏
   ↓
9. EventBus.start() 通知前端
   ↓
10. 前端跳转到主界面（index.html）
    ↓
11. 运行游戏
```

### 13.9 玩家模式实现（第二阶段）

**核心架构**:
```
┌─────────────────────────────────────────────────────────────┐
│  玩家模式架构                                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  真人玩家 3 号                                                │
│     │                                                       │
│     ├─→ 查看自己的身份（预言家）                             │
│     │   └─→ GET /api/game/1234/player/3/view               │
│     │                                                       │
│     ├─→ 获取 Agent 建议                                       │
│     │   └─→ GET /api/game/1234/player/3/suggestion         │
│     │       ← 返回：局势分析 + 建议发言                       │
│     │                                                       │
│     ├─→ 决定发言内容                                         │
│     │   ├─→ 采纳建议（一键使用）                             │
│     │   ├─→ 修改建议（编辑后发送）                           │
│     │   └─→ 自己输入（忽略建议）                             │
│     │   └─→ POST /api/game/1234/player/3/speech            │
│     │                                                       │
│     └─→ 发言显示在聊天窗口（其他玩家可见）                   │
│                                                             │
│  隐私保护：                                                  │
│  - 玩家 3 号只能看到自己的 Agent 建议                            │
│  - 玩家 3 号看不到其他玩家的真实身份                           │
│  - 其他玩家看不到玩家 3 号的 Agent 建议                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Agent 建议生成流程**:
```
1. 轮到玩家发言时
   ↓
2. Orchestrator 调用 Agent.get_suggestion(context)
   ↓
3. Agent 分析局势：
   - 查看已发生的事件（发言、投票、死亡）
   - 分析各玩家行为
   - 结合自己的身份和已知信息
   ↓
4. 生成建议：
   - 局势分析文本
   - 建议发言内容
   - 置信度评分
   - 备选方案
   ↓
5. 返回给前端显示
```

**玩家发言处理**:
```
1. 玩家点击"采纳建议"或"发送"
   ↓
2. POST /api/game/<game_id>/player/<player_id>/speech
   Body: {"content": "我是预言家..."}
   ↓
3. Web Server 验证玩家身份
   ↓
4. 调用 Orchestrator.submit_speech(player_id, content)
   ↓
5. 发言推送到 EventBus
   ↓
6. 所有玩家（上帝视角）可见
```

**夜晚行动处理（玩家模式）**:
```
1. 夜晚阶段开始（预言家行动）
   ↓
2. 前端显示行动 UI（选择查验目标）
   ↓
3. 玩家获取 Agent 建议
   GET /api/game/1234/player/3/suggestion?phase=night
   ← 返回：{"action": "check", "target": 5, "reason": "..."}
   ↓
4. 玩家决定：
   - 采纳建议（点击 5 号）
   - 自己选择（点击其他玩家）
   ↓
5. 提交行动
   POST /api/game/1234/player/3/action
   Body: {"action": "check", "target": 1}
   ↓
6. Orchestrator 接收行动，继续游戏流程
```

**会话管理（简化方案）**:

# 第一阶段：简单 token 验证
class PlayerSession:
    def __init__(self):
        self.sessions = {}  # {token: player_id}
    
    def create_session(self, player_id: int) -> str:
        """创建会话"""
        token = generate_token()
        self.sessions[token] = player_id
        return token
    
    def verify_session(self, token: str) -> Optional[int]:
        """验证会话"""
        return self.sessions.get(token)

# 配置页面选择玩家模式时生成 token

GET /api/game/1234/player/3/suggestion
Headers: {"Authorization": "Bearer <token>"}
```

**前端玩家界面组件**:
```javascript
// PlayerView.jsx - 玩家专属界面
function PlayerView({ gameId, playerId, token }) {
  const [myRole, setMyRole] = useState(null);
  const [suggestion, setSuggestion] = useState(null);
  const [speech, setSpeech] = useState('');
  
  // 获取自己的身份
  useEffect(() => {
    fetch(`/api/game/${gameId}/player/${playerId}/view`, {
      headers: { 'Authorization': `Bearer ${token}` }
    })
    .then(res => res.json())
    .then(data => setMyRole(data.my_role));
  }, []);
  
  // 获取 Agent 建议
  useEffect(() => {
    fetch(`/api/game/${gameId}/player/${playerId}/suggestion`, {
      headers: { 'Authorization': `Bearer ${token}` }
    })
    .then(res => res.json())
    .then(data => setSuggestion(data));
  }, []);
  
  // 采纳建议
  const adoptSuggestion = () => {
    setSpeech(suggestion.recommended_speech);
  };
  
  // 提交发言
  const submitSpeech = () => {
    fetch(`/api/game/${gameId}/player/${playerId}/speech`, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${token}`
      },
      body: JSON.stringify({ content: speech })
    });
  };
  
  return (
    <div className="player-view">
      <div className="my-info">
        <h3>你 ({playerId}号)</h3>
        <p>身份：{myRole}</p>
      </div>
      
      <div className="agent-suggestion">
        <h4>Agent 建议</h4>
        <p>{suggestion?.analysis}</p>
        <blockquote>{suggestion?.recommended_speech}</blockquote>
        <button onClick={adoptSuggestion}>采纳建议</button>
      </div>
      
      <div className="speech-input">
        <textarea
          value={speech}
          onChange={e => setSpeech(e.target.value)}
          placeholder="输入发言内容..."
        />
        <button onClick={submitSpeech}>发送</button>
      </div>
    </div>
  );
}
```
