# 狼人杀 UI 需求草案

**版本**: v2.0  
**更新日期**: 2026-03-25  
**状态**: 待讨论  
**定位**: 未来 UI 方案草案，不代表当前代码已实现

---

## 1. 文档定位

这份文档用于记录狼人杀 UI 的未来需求与边界，方便后续单独讨论 UI 方案。

它不是当前项目的实现说明文档。当前狼人杀真实可运行入口仍然是：

- `werewolf_main.py`

当前已落地的是 CLI 交互与复盘能力，不是 Web UI。

当前实现说明请以 [../refactor_202603/WEREWOLF_GUIDE.md](../refactor_202603/WEREWOLF_GUIDE.md) 为准。

---

## 2. 当前现状

当前项目中的狼人杀部分已经具备：

- CLI 观察模式
- CLI 玩家模式
- 游戏日志输出
- 复盘报告生成

当前项目中还没有：

- Web 服务器
- 浏览器端页面
- SSE / WebSocket 事件推送
- UI 专用 API 层
- `--no-ui`、`--config` 之类命令行参数

因此，后续 UI 设计必须以“新增能力”视角来规划，而不能假设这些能力已经存在。

---

## 3. UI 建设目标

狼人杀 UI 后续希望解决的问题：

1. 让开发者能以上帝视角观察一整局 AI 对战
2. 让测试者更容易查看发言、投票、死亡、技能与复盘信息
3. 为未来真人参与模式预留交互空间

---

## 4. 推荐分阶段范围

### 第一阶段：上帝视角调试 UI

目标：

- 仅服务开发和测试
- 后端必须可以脱离前端独立运行
- 前端只是可选观察层，不能成为游戏主流程的运行前提
- 不直接改写现有游戏规则与主流程
- 允许新增事件适配层、状态快照层和输入适配层
- 以“展示现有状态和日志”为主，服务调试游戏逻辑

建议包含：

- 游戏状态栏
- 玩家列表与身份显示
- 发言时间线
- 投票结果展示
- 夜晚行动结果展示
- 复盘报告入口

### 第二阶段：玩家参与 UI

目标：

- 支持真人以某个座位参与游戏
- 在轮到真人发言、投票、技能决策时提供输入界面
- 允许系统提供 Agent 建议，但最终由真人确认

建议包含：

- 玩家视角身份面板
- 发言输入框
- 投票与技能操作面板
- Agent 建议区

#### 玩家模式的目标边界

玩家模式的目标不是做多人联机房间系统，而是让一个真人替代某个固定座位参与单局狼人杀。

建议边界：

- 一局只接入一个真人玩家
- 真人在开局前绑定固定座位
- 除该真人外，其余玩家继续由 Agent 扮演
- 游戏主流程仍由后端自动推进

#### 玩家模式的完整交互流程

建议按以下流程运行：

1. 创建对局时指定 `mode=player`
2. 同时指定 `human_player_id`
3. 后端初始化完成后自动开局
4. UI 按玩家视角轮询状态并补拉事件
5. 当出现属于该真人的 `pending_input` 时，显示输入面板
6. 真人输入完成后提交，后端继续推进游戏
7. 非真人决策节点一律自动运行
8. 游戏结束后查看结果与复盘状态

#### Agent 建议的作用

Agent 建议建议作为“辅助参考”而不是强制动作。

推荐规则：

- 只在真人即将输入时生成
- 与当前 `pending_input` 绑定
- 支持三种使用方式：
  - 直接采纳
  - 修改后提交
  - 完全忽略
- 如果建议生成失败，不应阻塞真人输入

#### 玩家模式与狼人杀规则的关系

玩家模式不得改变当前狼人杀核心规则，只改变输入来源。

具体约束：

- 不能让真人在不属于自己的阶段提交动作
- 不能让真人跨阶段修改已生效结果
- 不能因为 UI 便利而合并原本分开的规则节点
- 不能让真人看到本不属于自己的隐藏信息

#### 玩家模式的通信要求

第一阶段玩家模式建议继续沿用：

- 状态轮询
- 事件补拉

暂不要求实时推送作为前提。

#### 玩家模式输入示例

为了让后续实现更直观，建议至少先明确三类真人输入在 UI 里的最小表现形式。

1. 发言输入

```json
{
  "request_id": "req_speech_001",
  "input_type": "speech",
  "player_id": 3,
  "phase": "day_discussion",
  "prompt": "请进行发言",
  "options": null,
  "timeout": 90,
  "metadata": {
    "round": 1
  }
}
```

建议 UI 行为：

- 展示文本输入框
- 可选择采纳 Agent 建议、修改后提交或自行输入
- 超时后允许空发言或记为未发言

2. 投票输入

```json
{
  "request_id": "req_vote_001",
  "input_type": "vote",
  "player_id": 3,
  "phase": "day_vote",
  "prompt": "请选择投票对象",
  "options": [1, 2, 4],
  "timeout": 60,
  "metadata": {
    "allow_abstain": true
  }
}
```

建议 UI 行为：

- 展示候选玩家列表
- 明确当前是否允许弃票
- 提交后立即退出等待状态

3. 技能输入

```json
{
  "request_id": "req_skill_001",
  "input_type": "skill",
  "player_id": 3,
  "phase": "night_seer",
  "prompt": "请选择要查验的玩家",
  "options": [1, 2, 4, 5, 6],
  "timeout": 60,
  "metadata": {
    "skill_name": "seer_check",
    "allow_skip": false
  }
}
```

建议 UI 行为：

- 展示技能名称和候选目标
- 明确该技能是否允许跳过
- 不允许选择规则外目标

这些输入示例的重点不是固定死 UI 样式，而是让后端契约先稳定下来。

## 6. UI 与现有代码的关系

未来 UI 必须建立在当前代码之上，优先复用：

- `games/werewolf/orchestrator.py`
- `games/werewolf/state.py`
- `services/logger_service.py`
- `services/game_review_service.py`

推荐原则：

1. 后端是主系统，前端是可选壳
2. 不让 UI 直接改写核心游戏规则
3. UI 只消费后端暴露的状态和事件
4. 权限隔离必须在后端完成，而不是由前端自行隐藏
5. 若需要 Web 支持，应新增适配层，而不是把 Web 逻辑直接塞进核心模块
6. 日志系统只作为调试辅助，不应直接充当 UI 正式接口

### 6.1 推荐分层

建议未来实现时按三层组织：

- 核心层
  - 游戏规则
  - 状态推进
  - Agent 决策
  - 复盘逻辑
- 适配层
  - 状态快照序列化
  - 事件流封装
  - 输入适配接口
  - 未来 Web / CLI 连接层
- 展示层
  - CLI
  - 未来 Web UI

### 6.2 明确约束

- 没有前端时，后端必须仍然能完整跑通一局游戏
- 未来 UI 不应成为游戏主流程的运行前提
- 视角隔离必须由后端决定，不依赖前端自觉过滤
- 如果以后需要 Web 服务，应优先新增独立适配代码，而不是直接侵入 `orchestrator`、`state` 和角色 Agent

## 7. UI 最小可行数据面

如果未来做 UI，建议先明确区分不同视角的数据边界。

### 7.1 上帝视角数据面

适用于开发者和测试者，允许看到完整信息：

- 当前游戏 ID
- 天数 / 夜数
- 当前阶段
- 存活玩家列表
- 玩家身份、阵营与死亡状态
- 警长状态
- 发言事件
- 投票事件
- 夜晚行动结果
- 游戏结束结果
- 复盘状态
- 复盘摘要
- 复盘文件路径

### 7.2 玩家视角数据面

适用于未来真人参与模式，必须做信息隔离：

- 当前游戏 ID
- 天数 / 夜数
- 当前阶段
- 自己的身份与状态
- 公共聊天事件
- 公共投票结果
- 自己可执行的操作
- 面向自己的 Agent 建议

玩家视角默认不应直接暴露：

- 其他玩家真实身份
- 其他玩家内心信息
- 未公开的夜晚决策过程
- 上帝视角专用日志内容

这些数据大部分已经能从当前状态对象和日志系统中提取，但还没有专门的 UI 适配接口。

### 7.3 视角隔离原则

视角隔离建议作为后端硬边界处理，而不是前端显示规则。

核心原则：

1. 同一局游戏可以有多个视图，但每个视图只能拿到自己被允许看到的数据
2. 状态快照和事件流都必须做可见性裁剪
3. Agent 建议属于本人私有信息，不进入公共数据面
4. 拿不准是否应该暴露的数据，第一阶段默认不暴露

### 7.4 视图上下文

后端在生成状态快照或事件流时，建议显式携带视图上下文：

- `view_type`
- `viewer_player_id`
- `is_debug_view`

推荐约定：

- 上帝视角：`view_type=god`，`viewer_player_id=null`
- 玩家视角：`view_type=player`，`viewer_player_id=<seat_id>`

### 7.5 状态裁剪规则

状态快照必须按视图裁剪，而不是把完整状态交给前端自行隐藏。

建议规则：

- 上帝视角可读取完整玩家身份、阵营、夜晚信息和调试字段
- 玩家视角只能读取公开状态，加上本人私有状态
- `players` 列表中的 `role`、`camp` 等敏感字段必须按视图裁剪
- `pending_input` 只能对当前被请求输入的玩家可见；上帝视角可见
- Agent 建议不进入公共状态区

### 7.6 事件裁剪规则

事件流同样必须做可见性裁剪，不能只裁剪状态快照。

推荐事件可见性至少支持：

- `public`
- `god_only`
- `private_player`

如果事件是 `private_player`，建议额外携带：

- `target_player_id`

这样像“预言家收到查验结果”这样的事件，就只会被：

- 对应玩家视角看到
- 上帝视角看到

而不会泄露给其他玩家视角。

### 7.7 Agent 建议的可见性

Agent 建议建议明确按“本人私有信息”处理：

- 只对当前真人玩家本人可见
- 不进入公共事件流
- 不写入玩家共享状态区
- 如需调试查看，只允许上帝视角通过专门调试字段查看

### 7.8 后端实现建议

为了避免在业务代码里散落大量 `if god / if player` 判断，建议把视角裁剪放到单独投影层：

- `GodViewSerializer`
- `PlayerViewSerializer`
- `GodEventProjector`
- `PlayerEventProjector`

核心层继续产出原始状态和原始事件，视图层再负责安全投影。

### 7.9 第一阶段结论

第一阶段建议采用最保守策略：

- 上帝视角默认完整可见
- 玩家视角默认最小可见
- 任何没有明确讨论过的数据，默认不暴露给玩家视角

## 8. 状态模型

### 8.1 设计目标

状态模型用于回答一个问题：

- “此刻游戏整体处于什么状态”

它适合：

- 页面初始化
- 断线重连后恢复
- 定时轮询
- 调试面板整体刷新

状态模型不等价于事件流，也不应直接复用原始日志结构。

### 8.2 推荐的顶层状态快照

建议未来 UI 状态快照至少包含以下顶层字段：

- `game_id`
- `mode`
- `view_type`
- `lifecycle_status`
- `phase`
- `day_number`
- `night_number`
- `president_id`
- `alive_player_ids`
- `players`
- `pending_input`
- `review_status`
- `result`
- `meta`

### 8.3 顶层字段说明

#### `mode`

表示当前游戏运行模式，例如：

- `observer`
- `player`

#### `view_type`

表示当前返回的是哪种视角的状态：

- `god`
- `player`

#### `lifecycle_status`

建议统一描述一局游戏当前生命周期，例如：

- `idle`
- `created`
- `initializing`
- `running`
- `waiting_input`
- `finished`
- `review_running`
- `review_ready`
- `error`

#### `phase`

建议统一表达当前阶段，例如：

- `setup`
- `president_election_speech`
- `president_election_vote`
- `president_election_pk`
- `day_discussion`
- `day_vote`
- `day_vote_pk`
- `last_words`
- `night_guard`
- `night_wolf`
- `night_witch`
- `night_seer`
- `hunter_skill`
- `game_over`

建议说明：

- `phase` 优先表示“当前 UI 和外部接口需要感知的规则节点”
- 不建议直接复用内部 Agent prompt 用的所有 `context["phase"]` 值
- 如果内部实现需要更细粒度上下文，可继续放在 `metadata` 或内部上下文对象中，而不是污染外部状态契约

### 8.4 玩家状态结构

建议 `players` 字段返回列表或字典，但字段结构要稳定。单个玩家状态建议包含：

- `player_id`
- `display_name`
- `is_alive`
- `role`
- `public_role`
- `camp`
- `is_human`
- `personality`
- `death_cause`
- `has_last_words`
- `is_president`

其中：

- `role` 只应在上帝视角或本人视角可见
- `public_role` 用于表达公开可见身份，例如警长、已死亡、已明牌身份等
- `camp` 默认也不应对普通玩家视角公开

### 8.5 上帝视角与玩家视角的状态差异

#### 上帝视角状态

允许看到：

- 全部真实身份
- 阵营
- 完整死亡原因
- 当前完整游戏结果
- 复盘状态

#### 玩家视角状态

只允许看到：

- 自己真实身份
- 公开发言与公开投票
- 已公开的死亡与阶段信息
- 面向自己的待输入请求
- 面向自己的 Agent 建议

玩家视角默认不返回其他玩家的真实身份、隐藏阵营和夜间私有信息。

### 8.6 `pending_input`

如果当前没有等待输入，`pending_input` 应为 `null`。

如果当前需要输入，则 `pending_input` 至少包含：

- `request_id`
- `input_type`
- `player_id`
- `prompt`
- `options`
- `timeout`
- `metadata`

这样 UI 不需要自己猜当前是否该弹输入框。

### 8.7 `review_status`

建议独立建模，而不是只给一个报告路径。

可以考虑包含：

- `status`
- `summary`
- `paths`
- `error`

例如 `status` 可以是：

- `disabled`
- `pending`
- `running`
- `ready`
- `failed`

### 8.8 状态模型的约束

- 状态快照用于表达“当前状态”，不是历史记录
- 状态快照应可被前端完整替换
- 上帝视角状态和玩家视角状态应由后端分别生成
- 状态字段命名必须稳定，不能直接暴露内部 Python 对象细节

### 8.9 `lifecycle_status` / `phase` / `pending_input` 的关系

为了避免把“系统运行状态”和“狼人杀规则阶段”混淆，建议明确把这三个字段分开理解：

- `lifecycle_status`
  - 描述当前会话运行到什么状态
- `phase`
  - 描述当前狼人杀规则推进到哪个阶段
- `pending_input`
  - 描述当前是否正在等待真人输入，以及等什么输入

可以把它理解成：

```text
lifecycle_status = 运行层视角
phase            = 规则层视角
pending_input    = 交互层视角
```

#### 典型组合示例

1. 上帝模式白天讨论中

```json
{
  "lifecycle_status": "running",
  "phase": "day_discussion",
  "pending_input": null
}
```

含义：

- 游戏正在自动运行
- 当前处于白天发言阶段
- 不需要真人输入

2. 玩家模式等待真人投票

```json
{
  "lifecycle_status": "waiting_input",
  "phase": "day_vote",
  "pending_input": {
    "request_id": "req_vote_001",
    "input_type": "vote",
    "player_id": 3
  }
}
```

含义：

- 游戏当前暂停
- 当前规则阶段是白天投票
- 正在等待 3 号真人提交投票

3. 游戏结束，复盘生成中

```json
{
  "lifecycle_status": "review_running",
  "phase": "game_over",
  "pending_input": null
}
```

含义：

- 主对局已经结束
- 当前处于赛后复盘阶段
- 没有待输入操作

#### 组合约束

建议至少遵守这些约束：

- 当 `lifecycle_status=waiting_input` 时，`pending_input` 不应为 `null`
- 当 `pending_input` 不为 `null` 时，`lifecycle_status` 应为 `waiting_input`
- 上帝模式下，默认 `pending_input=null`
- 当 `lifecycle_status=finished`、`review_running`、`review_ready` 或 `error` 时，不应再接受普通玩家输入

这样前端就可以比较稳定地根据这三个字段判断：

- 页面该显示观察态还是等待态
- 当前是规则上的哪个阶段
- 是否需要渲染输入面板

### 8.10 `phase` 与当前狼人杀流程的映射

为了让文档和当前代码实现对齐，建议把对外 `phase` 枚举理解为“稳定的外部规则阶段”，而不是逐字照搬所有内部调用上下文。

推荐映射如下：

| 对外 `phase` | 当前代码中的主要流程节点 | 说明 |
|------|------|------|
| `setup` | 对局创建、玩家初始化 | 开局前准备阶段 |
| `president_election_speech` | `_run_president_election()` 候选人发言 | 警长竞选发言 |
| `president_election_vote` | `_run_president_election()` 首轮投票 | 警长竞选投票 |
| `president_election_pk` | `_handle_president_tie()` | 警长 PK 发言与再投票 |
| `day_discussion` | `_run_day_speech()` | 白天正常发言阶段 |
| `day_vote` | `_run_vote()` 首轮投票 | 白天放逐投票 |
| `day_vote_pk` | `_run_vote()` 平票 PK | 白天 PK 发言与再投票 |
| `last_words` | 放逐遗言、夜亡遗言 | 遗言阶段 |
| `night_guard` | `_run_guard_action()` | 守卫行动 |
| `night_wolf` | `_run_wolf_night()` | 狼人协商与袭击 |
| `night_witch` | `_run_witch_action()` | 女巫行动 |
| `night_seer` | `_run_seer_action()` | 预言家查验 |
| `hunter_skill` | `_handle_hunter_skill()` | 猎人开枪 |
| `game_over` | `_end_game()` 前后 | 游戏结束阶段 |

#### 设计原则

建议遵守以下原则：

- 对外 `phase` 要稳定，优先服务 UI 和接口契约
- 内部实现可以继续保留更细上下文，例如 `day_speech`、`voting`、`night_action`
- 如果后续规则节点继续细化，优先新增清晰的外部 `phase`，不要复用含义模糊的老字段

#### 为什么不直接暴露内部 `context["phase"]`

当前代码里内部上下文值更多是给 Agent 推理和日志使用，例如：

- `day_speech`
- `voting`
- `night_action`
- `president_election_voting`

这些值不一定适合作为 UI 的长期稳定契约，因为：

- 粒度混杂
- 同名上下文可能跨多个角色共用
- 后续 prompt 调整时容易变化

因此更推荐：

- 外部接口使用稳定 `phase`
- 内部上下文继续自由演进

## 9. 状态快照获取方式

### 9.1 推荐原则

第一阶段建议优先采用：

- 完整状态快照
- 前端整包替换
- 简单轮询或显式刷新

而不是一开始就做复杂的局部状态补丁。

原因是当前项目的主要目标仍然是：

- 保持后端可独立运行
- 方便调试游戏逻辑
- 降低 UI 接入复杂度

### 9.2 推荐方式

建议状态接口先支持“返回当前完整快照”。

也就是说，每次前端请求状态时，后端直接返回当前完整的视角化状态对象，而不是只返回变化字段。

这样做的优点是：

- 后端实现简单
- 前端不需要复杂合并逻辑
- 调试时更容易核对
- 断线恢复更直接

### 9.3 上帝视角下的建议策略

上帝视角第一阶段建议以“可读、稳定、便于调试”为优先目标，因此推荐：

- 初始进入页面时拉取一次完整状态
- 游戏运行中按固定频率轮询完整状态
- 事件流作为辅助，不依赖它来重建完整状态

### 9.4 玩家视角下的建议策略

玩家视角也建议先遵循同样原则：

- 初次进入时拉取完整视角状态
- 当需要用户输入或发生关键阶段切换时，允许主动刷新状态

后续如果玩家模式对实时性要求更高，再考虑在完整状态快照之外叠加更细粒度的实时更新。

### 9.5 更新频率建议

第一阶段可以使用比较保守的策略，例如：

- 页面初始化时立即拉取
- 游戏运行中每隔固定时间拉取一次
- 收到关键事件后额外触发一次刷新
- 用户提交输入后立即刷新一次

这样能兼顾实现简单和调试体验。

### 9.6 当前阶段不建议优先做的事情

当前阶段不建议把重点放在：

- 状态局部 patch
- 前端复杂状态合并
- 只靠事件流重建页面状态
- 过早优化网络开销

这些都更适合在第一阶段跑通之后再讨论。

### 9.7 结论

状态模型第一阶段建议采用：

- “完整快照 + 前端整包替换” 作为主方案
- 事件流只做增量提示和时间线展示
- 不要求前端自己拼装最终状态

## 10. 事件模型

### 10.1 设计目标

事件模型用于回答另一个问题：

- “刚刚发生了什么”

它适合：

- 聊天时间线
- 实时更新
- 调试回放
- 局部增量刷新

状态快照解决“现在是什么样”，事件流解决“刚刚发生了什么”，两者必须分开。

### 10.2 推荐的统一事件结构

建议所有 UI 事件都采用统一外层结构，至少包含：

- `event_id`
- `sequence`
- `timestamp`
- `game_id`
- `phase`
- `event_type`
- `visibility`
- `payload`

### 10.3 字段说明

#### `event_id`

事件唯一标识，用于去重和补拉。

#### `sequence`

单局内严格递增的顺序号，用于排序、断线续传和补拉。

#### `visibility`

建议明确可见范围，例如：

- `public`
- `god_only`
- `private_player`

如果是 `private_player`，还应在 `payload` 或扩展字段里标明对应 `target_player_id`。

### 10.4 最小事件集合

第一阶段建议至少支持以下事件类型：

- `game_created`
- `game_started`
- `phase_changed`
- `speech`
- `vote_cast`
- `vote_result`
- `night_action_result`
- `death`
- `president_changed`
- `input_requested`
- `input_received`
- `game_finished`
- `review_status_changed`
- `error`

### 10.5 事件与日志的关系

日志系统仍然可以继续保留，用于：

- 调试
- 文件留存
- 复盘

但 UI 事件不应直接复用日志原始结构。建议做单独转换层，把内部日志或状态变成稳定的 UI 事件 DTO。

### 10.6 事件模型的约束

- 每个事件必须有稳定顺序号
- 每个事件必须有可见性标记
- 事件类型命名要稳定
- 事件载荷要面向前端消费，而不是面向后端内部调试
- 上帝视角事件和玩家视角事件的可见范围必须由后端统一裁剪

## 11. 事件保留与断线恢复

### 11.1 推荐原则

由于第一阶段已经采用“完整状态快照”作为主方案，因此事件流不需要承担“恢复完整页面状态”的责任。

推荐职责划分如下：

- 状态快照负责恢复“现在的全局状态”
- 事件流负责补充“最近发生了什么”

### 11.2 断线恢复策略

建议断线恢复时优先执行以下步骤：

1. 重新拉取当前完整状态快照
2. 根据本地记录的最后事件顺序号，尝试补拉最近事件
3. 如果事件补拉失败或事件窗口已过期，仍以完整状态快照为准继续工作

这意味着：

- UI 不依赖事件流也能恢复
- 事件流丢失不会导致页面无法继续使用
- 后端实现压力更可控

### 11.3 推荐的事件补拉方式

建议事件接口支持基于顺序号补拉，例如：

- 给定 `last_sequence`
- 返回比它更新的事件列表

补拉接口的目标不是“全量回放整局历史”，而是“补最近一段缺失的事件”。

### 11.4 事件保留策略

第一阶段建议采用有限窗口缓存，而不是永久保留全部 UI 事件。

原因：

- UI 事件主要服务当前页面会话
- 全量历史更适合交给日志和复盘系统
- 有限窗口更容易控制内存和复杂度

推荐做法：

- 后端为每局游戏保留最近一段事件窗口
- 事件窗口用于页面补拉、时间线刷新和短时断线恢复
- 更长历史由日志文件和复盘系统负责

建议进一步明确：

- 事件缓存按“单局内最近 N 条事件”保留
- 第一阶段不强制写死具体数值，但建议默认从一个保守上限开始，例如数百到一千条量级
- 如果后续测试发现单局事件明显更多，再调整窗口上限，而不是一开始就做无限缓存

#### 建议清理策略

建议按以下规则清理：

- 新事件进入时，如果超过窗口上限，丢弃最旧事件
- 一局结束后，运行层可继续短暂保留该局事件窗口，供 UI 收尾查看
- 当创建下一局活动对局时，上一局 UI 事件缓存可以整体释放

#### 窗口过期时的约定

如果前端请求的 `last_sequence` 已经早于当前窗口起点，建议后端：

- 返回“事件窗口已过期”的明确标记
- 不尝试补全更早历史
- 引导前端重新拉完整状态并继续从最新窗口工作

这样可以避免事件接口承担历史归档职责。

### 11.5 与日志系统的分工

建议明确分工：

- UI 事件缓存
  - 面向实时展示
  - 有限窗口
  - 可按顺序号补拉
- 日志系统
  - 面向调试与持久化
  - 可保留更完整信息
  - 不直接作为前端事件接口
- 复盘系统
  - 面向赛后分析
  - 不参与实时 UI 状态恢复

### 11.6 前端建议行为

前端在第一阶段建议这样处理：

- 页面初始化：先拉完整状态
- 页面运行中：消费事件流并更新时间线
- 如果发现事件断档：重新拉完整状态，再尝试补拉事件
- 如果补拉失败：丢弃本地事件缓存，以最新状态快照为准

### 11.7 结论

第一阶段推荐方案是：

- 事件流采用有限窗口缓存
- 事件支持按顺序号补拉
- 断线恢复优先依赖完整状态快照
- 不要求事件流单独承担状态恢复职责

## 13. 输入适配层

### 13.1 设计目标

输入适配层的目标是让后端只表达“现在需要什么输入”，而不关心输入来自：

- CLI
- 未来 Web UI
- 自动化测试
- 预置脚本

也就是说，核心游戏逻辑不直接调用终端 `input()`，而是向统一输入接口发起请求。

### 13.2 推荐抽象

建议引入统一抽象，例如：

- `InputProvider`

它不负责推进游戏，只负责在后端发出输入请求时提供结果。

### 13.3 最小职责

`InputProvider` 至少应支持三类输入：

1. 发言输入
2. 投票输入
3. 技能输入

此外还应支持少量会话级输入，例如：

- 选择模式
- 选择真人座位

### 13.4 统一请求协议

建议不要为“发言 / 投票 / 技能”分别设计完全不同的调用方式，而是统一成一类输入请求对象。

例如，一个输入请求至少包含：

- `request_id`
- `game_id`
- `player_id`
- `phase`
- `input_type`
- `prompt`
- `options`
- `timeout`
- `metadata`

其中：

- `input_type` 可以是 `speech`、`vote`、`skill`、`mode_select`、`player_select`
- `options` 用于列出可选项
- `metadata` 用于补充技能上下文、候选人列表等信息

### 13.5 统一返回协议

输入结果也建议统一成一类响应对象，至少包含：

- `request_id`
- `game_id`
- `player_id`
- `submitted`
- `value`
- `raw_input`
- `metadata`

这样 CLI、UI、测试桩都可以用同一套方式返回结果。

### 13.6 不同模式下的行为

#### 上帝模式

- 不应产生玩家输入请求
- 游戏自动推进到结束

补充约束：

- 当前阶段先保持上帝模式默认连续运行，不引入实际暂停控制逻辑
- 但后端运行控制层应预留暂停 / 继续能力的扩展接口
- 这样后续如果需要做调试型 UI，可以在不改动核心规则层的前提下增加控制能力

#### 玩家模式

- 当轮到真人发言、投票或技能决策时，后端发起输入请求
- 只有这些节点会暂停等待输入
- 输入提交后，后端继续推进

#### 自动化测试模式

- 可以由测试输入提供器自动返回预设结果
- 不应依赖真实终端或页面输入

### 13.7 关键约束

- 输入适配层不应包含游戏规则判断
- 输入适配层不应决定游戏何时推进
- 输入适配层只负责“接收请求并返回结果”
- CLI 与未来 UI 只是在 `InputProvider` 上的两种实现

### 13.8 真人超时、取消与默认行为

这部分必须服从狼人杀规则流程，不能因为 UI 接入而让后端进入不确定状态。

建议第一阶段采用保守策略：

- 上帝模式下不涉及真人输入，因此不存在超时问题
- 玩家模式下，如果轮到真人输入，后端进入 `waiting_input`
- 第一阶段默认不自动替真人做高风险决定
- 第一阶段不支持“取消当前回合并回退游戏”

针对不同输入类型，建议这样处理：

#### 发言输入

- 允许超时后提交空发言或由系统记录“未发言”
- 不应因为真人超时而阻塞整局无限挂起

#### 投票输入

- 建议支持明确的默认行为
- 默认行为可选为“弃票”或“按规则记为未投票”
- 最终采用哪种方式必须与当前狼人杀规则实现保持一致

#### 技能输入

- 默认不应替真人做主动技能决策
- 若超时，建议按“放弃使用技能”处理
- 但必须确保不违反当前角色规则，例如只能在该角色本轮本来就可放弃时这样做

#### 取消输入

- 第一阶段建议只允许“重新填写当前输入”，不允许撤销已经提交并生效的输入
- 一旦提交成功，后端即可继续推进，前端不再拥有回滚权

### 13.9 输入请求与事件的关联

为了让 UI 更稳定地处理等待与提交，建议明确：

- 每个输入请求必须有唯一 `request_id`
- `input_requested` 事件中的 `request_id` 必须与状态中的 `pending_input.request_id` 一致
- `input_received` 事件必须引用同一个 `request_id`
- 输入接口只能接受当前仍有效的 `request_id`

这样可以避免：

- 页面重复提交
- 旧输入误投到新阶段
- 事件时间线与状态快照对不上

## 14. 游戏驱动方式

### 14.1 推荐原则

后续建议采用“后端运行控制器驱动，CLI / UI 只是外层接入”的方式。

也就是说：

- 不是 CLI 驱动游戏
- 也不是未来 UI 驱动游戏
- 真正驱动游戏推进的是后端统一的运行控制器

CLI 和未来 UI 的职责只是：

- 发起一局游戏
- 提供配置
- 在需要真人输入时提交输入
- 消费状态和事件

### 14.2 推荐结构

建议未来把运行过程分成三层：

- 编排器层
  - 只负责游戏规则与流程编排
  - 不直接处理终端输入、页面输入或 HTTP 交互
- 运行控制层
  - 负责启动一局游戏
  - 管理运行状态
  - 决定什么时候自动推进
  - 决定什么时候等待真人输入
  - 对外暴露状态快照与事件流
- 接入层
  - CLI 接入
  - 未来 Web UI 接入
  - 测试接入

### 14.3 自动推进与阻塞点

建议把一局游戏中的节点分成两类：

#### 自动推进节点

- 初始化游戏
- AI 发言
- AI 投票
- AI 夜晚行动
- 结算夜晚结果
- 胜负判断
- 复盘任务触发

这些节点默认由后端自动推进，不需要前端参与。

#### 等待输入节点

- 选择游戏模式
- 选择真人玩家编号
- 真人发言
- 真人投票
- 真人技能选择
- 未来可能的调试控制操作（暂停 / 继续 / 单步）

这些节点需要通过统一输入接口等待外部输入。

### 14.4 明确暂停规则

无论未来接入的是 CLI 还是 UI，都遵循同一条规则：

- 只有在确实需要用户输入时，系统才暂停等待输入

对应地：

- 上帝模式下，整局游戏应自动推进，不因 UI 或 CLI 的存在而额外阻塞
- 当前阶段先不引入上帝模式暂停逻辑，但后端应预留未来扩展点
- 玩家模式下，只在轮到真人发言、投票、使用技能，或进行必要模式选择时暂停
- 如果当前节点不需要真人决策，后端应继续自动运行，不等待前端确认

这意味着 CLI 和 UI 都不负责“推动每一步继续执行”，它们只在后端发出“需要输入”信号时响应。

### 14.5 对当前代码的演进方向

当前 `werewolf_main.py` 仍然直接：

- 读取配置
- 询问模式
- 询问复盘参数
- 通过 `input()` 获取真人输入
- 直接运行 `WerewolfOrchestrator`

后续建议演进为：

1. 保留 `WerewolfOrchestrator` 作为核心游戏编排器
2. 新增统一运行控制器，负责管理对局生命周期
3. 把 `input()` 相关逻辑逐步移出主流程
4. 让 CLI 与未来 UI 都通过同一套输入适配接口与运行控制器交互

### 14.6 关键约束

- 后端必须能在没有 UI 的情况下完整自动运行
- UI 不应直接调用编排器内部方法来逐步推进游戏
- 游戏推进节奏必须由后端统一控制
- 未来即使做暂停、继续或单步调试，也应由运行控制层提供能力，而不是由前端自行拼装流程

### 14.7 调试控制的扩展方向

调试能力主要服务上帝视角，不应默认渗透到玩家模式。

后续如果需要增强调试体验，建议优先考虑这些能力：

- 暂停
- 继续
- 单步推进
- 查看更详细的调试日志
- 查看 Agent 决策上下文

建议约束：

- 玩家模式默认不展示调试字段
- 玩家模式默认不提供暂停 / 单步推进
- 玩家模式默认不展示其他角色的内心信息与隐藏日志
- 这些能力都应建立在运行控制层之上，而不是由 UI 自己拼接流程

## 15. 通信方案

### 15.1 第一阶段推荐方案

第一阶段推荐采用：

- 状态轮询
- 事件补拉

而不是一开始就上 SSE 或 WebSocket。

### 15.2 为什么优先这样做

原因主要有四个：

1. 当前项目的核心目标仍然是调试游戏逻辑，而不是构建高实时交互系统
2. 我们已经确定“完整状态快照”是主方案，因此没有必要让前端高度依赖实时推送
3. 轮询 + 补拉实现简单、调试直接、故障面更小
4. 这套方案更符合“后端可独立运行、前端只是可选层”的总原则

### 15.3 推荐组合方式

建议第一阶段按以下方式协同：

#### 状态接口

用于获取当前完整快照：

- 页面初始化时调用
- 断线恢复时调用
- 关键操作后调用
- 定时轮询调用

#### 事件接口

用于按顺序号补拉最近事件：

- 获取最近发言、投票、阶段切换等时间线变化
- 辅助前端更新显示
- 不负责重建完整状态

### 15.4 推荐的前端工作流

第一阶段前端建议这样工作：

1. 页面加载时拉取完整状态
2. 记录当前最后事件顺序号
3. 周期性轮询完整状态
4. 周期性或按需补拉最近事件
5. 如果事件补拉失败，直接以最新完整状态为准

### 15.5 为什么不优先使用 SSE

SSE 不是不能做，而是当前阶段不建议优先：

- 会增加服务端连接管理复杂度
- 会让页面更依赖持续连接
- 对当前“调试优先”的目标帮助没有状态快照那么大
- 一旦事件协议尚未稳定，推送链路只会放大调试成本

SSE 更适合在以下条件成熟后再考虑：

- 状态模型已经稳定
- 事件模型已经稳定
- 确认需要更强的实时体验

### 15.6 为什么不优先使用 WebSocket

WebSocket 当前更不建议优先：

- 它更适合强交互、多人并发、低延迟场景
- 当前项目还没有进入这类复杂阶段
- 对上帝视角调试 UI 来说，收益明显小于实现成本

### 15.7 第二阶段的演进策略

第一阶段和第二阶段不必强行绑定同一通信方案。

建议演进方式是：

- 第一阶段：状态轮询 + 事件补拉
- 第二阶段：如果玩家模式对实时交互要求更高，再评估是否引入 SSE
- 更后续阶段：如果出现多人并发、实时双向协作，再讨论 WebSocket

#### 建议的升级条件

不建议仅仅因为“看起来更先进”就从轮询升级到推送。

更合理的升级条件是同时满足以下一项或多项：

- 玩家模式下，真人频繁感知到轮询延迟影响操作体验
- 后台页需要更及时地展示高频事件，而轮询频率继续提高已经不经济
- 同一时刻存在多个观察端或接入端，需要更高效地分发事件
- 后续进入多人协作或更强交互场景

如果只是单机调试和单真人参与，优先继续使用轮询 + 事件补拉即可。

### 15.8 结论

当前最适合项目现状的方案是：

- 以完整状态轮询为主
- 以事件补拉为辅
- 不把 SSE / WebSocket 作为第一阶段前提

这样最有利于：

- 快速落地
- 后端独立调试
- 降低系统复杂度
- 后续平滑升级

### 15.9 是否支持局部状态更新

第一阶段建议明确：

- 不支持局部状态 patch 作为主方案
- 状态接口始终返回完整快照
- 前端以整包替换为主

原因：

- 更符合当前调试优先目标
- 更容易核对状态与事件是否一致
- 更不容易因为局部合并失败导致显示错误

后续只有在以下条件同时出现时，才建议考虑局部状态更新：

- 完整状态明显过大，轮询成本已成为实际瓶颈
- MVP 和第二阶段接口已经稳定
- 前端确实需要更细粒度刷新，而不仅是事件时间线更新

即便后续支持局部更新，也建议作为“增量优化能力”，而不是替代完整状态快照。

## 16. 待讨论问题

后续正式做 UI 前，建议先定以下问题：

1. UI 是桌面本地页、内嵌网页，还是独立 Web 服务
2. 第二阶段是否在现有方案上增加推送能力
3. 玩家模式中真人输入如何注入到当前 orchestrator
4. 上帝视角是否默认展示全部身份与内心信息，以及是否展示更深的调试上下文

本轮已经先收敛的默认方案：

- 第一阶段玩家模式遵守现有狼人杀规则，只替换真人输入来源
- 第一阶段复盘展示以状态、摘要和文件路径为主，不要求 UI 内嵌完整报告渲染
- 第一阶段技术路线继续优先“后端契约先行，再做只读调试页 MVP”

### 第一阶段只读调试页 MVP

第一阶段 MVP 建议明确定位为：

- 上帝视角游戏后台页
- 游戏运行监控页
- 调试观察页

它的目标不是给玩家直接对战使用，而是帮助开发和测试确认：

- 游戏是否正常运行
- 狼人杀规则是否按预期推进
- 当前是否卡在某个状态
- 最近发生了什么
- 游戏结束后复盘是否正常产出

#### MVP 必须包含

- 顶部状态栏
  - `game_id`
  - `lifecycle_status`
  - `phase`
  - 天数 / 夜数
  - 当前警长
  - 存活人数
- 玩家列表
  - 座位号
  - 存活 / 死亡
  - 真实身份
  - 阵营
  - 是否警长
- 事件时间线
  - 阶段切换
  - 发言
  - 投票
  - 夜晚行动结果
  - 死亡
  - 游戏结束
- 复盘状态区
  - 是否生成中
  - 是否完成
  - 报告路径
- 自动刷新
  - 轮询状态
  - 补拉最近事件

#### MVP 建议包含

- 当前 `pending_input` 显示
- `error` 状态和错误信息展示
- 简单事件过滤

#### MVP 第一版明确不做

- 登录与权限系统
- 多房间与多局并发
- 页面内嵌完整复盘报告
- 玩家输入面板
- Agent 建议展示
- 复杂图表
- 暂停 / 单步控制按钮
- SSE / WebSocket

#### MVP 成功标准

- 上帝模式下一局游戏可以从创建到结束完整展示
- 页面刷新后能够恢复到当前状态
- 事件时间线顺序稳定
- 状态与事件能基本对齐
- 游戏结束后能看到复盘状态
- 出错时能在页面中直接看到错误信息

#### MVP 推荐页面布局

1. 顶部：全局状态栏
2. 左侧：玩家面板
3. 右侧：事件时间线
4. 底部或侧边：复盘状态与错误信息

### 推荐目录结构

目录设计建议优先满足三件事：

1. 不侵入现有狼人杀核心逻辑
2. 让运行控制层和 UI 接入层职责分开
3. 为后续 CLI / Web 统一接入预留空间

推荐结构：

```text

  game_review_service.py

  werewolf/
    session.py
    controller.py
    input_provider.py
    event_store.py
    serializers.py
    errors.py

interfaces/
  cli/
    werewolf_cli.py
  web/
    app.py
    routes/
      session_routes.py
      state_routes.py
```

推荐职责：

- `games/werewolf/`
  - 继续只放狼人杀规则、状态和编排逻辑

  - 放运行控制层、会话对象、事件缓存、输入适配抽象、状态序列化

- `interfaces/web/`
  - 放 Web API、请求路由和接口结构

```text
  app.py
```

这样既能快速落地，也不会过早把目录拆得过重。

## 17. 启动与会话模型

### 17.1 第一阶段推荐方案

第一阶段建议采用：

- 单进程
- 单活动对局

也就是说，一个后端进程同一时间只维护一局狼人杀。

这样做的原因是：

- 最符合当前“单机调试优先”的目标
- 实现与调试成本最低
- 可以避免过早引入房间、多局调度、会话路由等复杂问题

### 17.2 启动模型

建议区分两种接入方式，但底层共用同一套运行控制器：

#### CLI 直跑模式

- 直接启动一局游戏
- 可以选择观察模式或玩家模式
- 没有前端也能完整跑通

#### UI 接入模式

- 先启动后端运行服务
- 再由 UI 发起“创建并启动当前对局”
- 后端仍只维护一局活动对局

### 17.3 配置分层

建议把配置分成两层：

#### 进程级配置

由后端启动时加载，例如来自：

- `config/system.json`

主要包括：

- LLM 配置
- 日志配置
- 默认游戏配置来源

#### 对局级配置

由创建一局游戏时确定，例如：

- `mode`
- `human_player_id`
- `review_enabled`
- `review_mode`
- `detect_loopholes`
- 可选的玩家人数或角色配置

这样可以避免把系统级配置和单局参数混在一起。

### 17.4 生命周期模型

建议一局游戏使用统一生命周期字段，例如：

- `idle`
- `created`
- `initializing`
- `running`
- `waiting_input`
- `finished`
- `review_running`
- `review_ready`
- `error`

推荐含义如下：

#### `idle`

当前没有活动对局。

#### `created`

对局配置已生成，但尚未正式开始。

#### `initializing`

正在创建编排器、初始化玩家和游戏状态。

#### `running`

游戏自动推进中。

#### `waiting_input`

当前暂停，等待真人输入。

#### `finished`

游戏胜负已确定，主流程已结束。

#### `review_running`

复盘任务正在进行。

#### `review_ready`

复盘结果已可读取。

#### `error`

本局进入异常状态。

#### 推荐状态迁移关系

建议第一阶段按以下迁移关系实现：

- `idle -> created`
- `created -> initializing`
- `initializing -> running`
- `running -> waiting_input`
- `waiting_input -> running`
- `running -> finished`
- `finished -> review_running`
- `finished -> review_ready`
- `review_running -> review_ready`
- `any_state -> error`

其中：

- `finished -> review_ready` 适用于未启用复盘，或复盘结果可立即读取的情况
- `any_state -> error` 表示任意阶段出现不可恢复异常时，都可以转入错误态

#### 不建议出现的迁移

为了避免生命周期失真，建议明确不支持以下跳转：

- `idle -> running`
- `waiting_input -> finished`
- `finished -> running`
- `review_ready -> running`

如果将来确实需要这些能力，也应通过新增明确状态或单独控制逻辑实现，而不是直接复用现有状态。

#### 生命周期与狼人杀规则阶段的关系

建议把生命周期状态和游戏规则阶段分开建模：

- `lifecycle_status`
- `phase`

例如：

- `lifecycle_status=running`, `phase=day_discussion`
- `lifecycle_status=waiting_input`, `phase=night_seer`
- `lifecycle_status=finished`, `phase=game_over`

这样可以避免把“系统是否在等待输入”和“当前是狼人杀哪一阶段”混在同一个字段里。

#### `waiting_input` 的进入规则

`waiting_input` 只应在这些节点进入：

- 真人发言
- 真人投票
- 真人技能决策
- 少量会话级选择

上帝模式默认不进入 `waiting_input`。

#### `error` 状态的建议结构

为了方便 UI 正确展示错误，建议错误状态至少携带：

- `code`
- `message`
- `stage`
- `recoverable`

这样前端才能区分：

- 只提示错误信息
- 允许重新创建一局
- 允许重试某些非核心流程，例如复盘生成

#### 简化状态迁移图

```text
idle
  -> created
  -> initializing
  -> running
     -> waiting_input
        -> running
     -> finished
        -> review_running
           -> review_ready

any_state -> error
finished -> review_ready
```

### 17.5 会话对象建议

未来运行控制层建议维护一个“当前活动对局容器”，至少包含：

- `session_id`
- `game_id`
- `config`
- `lifecycle_status`
- `orchestrator`
- `pending_input`
- `last_sequence`
- `created_at`
- `finished_at`
- `review_status`

其中：

- `session_id` 更偏运行层概念
- `game_id` 更偏游戏本身概念

第一阶段两者可以接近，但最好不要混成完全同一个语义。

### 17.6 UI 侧的最小动作集

如果按本方案实现，第一阶段 UI 只需要支持这些动作：

1. 查询当前是否存在活动对局
2. 如果没有，则创建一局新对局
3. 启动当前对局
4. 拉取完整状态
5. 拉取最近事件
6. 如果有 `pending_input`，提交输入
7. 查询复盘状态或读取复盘结果

也就是说，UI 不负责直接控制每一步游戏推进。

### 17.7 第一阶段非目标

第一阶段明确不做：

- 多局并发
- 房间系统
- 多人联机
- 复杂会话路由
- 前端隐式自动创建并启动游戏

### 17.8 结论

当前最适合项目现状的方案是：

- 单进程
- 单活动对局
- 进程级配置与对局级配置分离
- UI 负责创建 / 启动 / 查询 / 输入 / 查看结果
- 游戏推进仍由后端统一控制

## 18. 建议的下一步

如果后续要认真推进 UI，建议顺序是：

1. 按目录方案落第一版运行控制层骨架
2. 先补最小后端接口：创建、启动、状态、事件、输入、复盘状态
3. 做只读调试页 MVP，验证上帝视角后台页是否能稳定观察一局游戏
4. 再评估是否进入玩家模式输入页和更强实时能力

到这个阶段，这份文档已经可以作为第一版实现准备文档使用；后续若有变更，应优先回写契约与边界，而不是只改页面方案。

### 18.1 实现前检查清单

在真正开始写第一版 UI 相关代码前，建议先完成以下检查：

- 确认 `games/werewolf/` 中不直接引入 Web 或页面依赖
- 确认运行控制层放在 `runtime/werewolf/`
- 确认第一阶段只实现单进程、单活动对局
- 确认状态接口返回完整快照而不是局部 patch
- 确认事件接口只负责最近事件补拉，不承担历史归档
- 确认所有玩家视角数据都经过后端裁剪
- 确认输入请求统一使用 `request_id`
- 确认错误返回统一使用 `ok / data / error / meta`
- 确认 MVP 页面定位为上帝视角后台页，而不是玩家前台页
- 确认 UI 不会改变狼人杀原始规则推进顺序

如果这些前提没有满足，建议先补后端契约和运行层，再开始页面实现。

## 19. 第一阶段最小后端 API 边界

### 19.1 设计目标

第一阶段 API 的目标不是做成完整游戏平台接口，而是为“单机调试型 UI”提供最小可用后端能力。

因此第一阶段 API 应遵循：

- 足够少
- 足够稳
- 足够清晰
- 便于调试

### 19.2 第一阶段必须接口

建议第一阶段至少包含以下接口能力。

#### 1. 查询当前活动对局

用途：

- 判断当前是否已有活动对局
- 给 UI 决定显示“创建游戏”还是“查看当前对局”

建议职责：

- 返回当前是否存在活动对局
- 返回当前 `session_id` / `game_id`
- 返回当前生命周期状态

#### 2. 创建对局

用途：

- 根据对局级配置创建一局新游戏

建议职责：

- 接收对局级配置
- 创建活动对局
- 将生命周期置为 `created`

说明：

- 第一阶段由于是单活动对局，如果当前已有活动对局，应返回明确错误或拒绝创建

#### 3. 启动对局

用途：

- 让已创建但未开始的对局正式进入运行

建议职责：

- 将生命周期从 `created` 推进到 `initializing / running`
- 异步启动后端运行控制器

#### 4. 获取当前完整状态

用途：

- 页面初始化
- 页面轮询
- 断线恢复
- 提交输入后刷新

建议职责：

- 按请求视角返回完整状态快照
- 支持上帝视角
- 未来可扩展玩家视角

#### 5. 获取最近事件

用途：

- 更新时间线
- 进行短时补拉

建议职责：

- 接收 `last_sequence`
- 返回之后的新事件列表

#### 6. 提交输入

用途：

- 当 `pending_input` 存在时，由 CLI 或 UI 提交输入结果

建议职责：

- 校验 `request_id`
- 校验输入是否属于当前等待节点
- 提交结果给运行控制层
- 成功后推动游戏继续运行

#### 7. 查询复盘状态

用途：

- 游戏结束后查看复盘进度与结果

建议职责：

- 返回 `review_status`
- 可返回摘要与文件路径

### 19.3 第一阶段可选接口

以下接口不是第一阶段必须项，但后续可能有价值：

#### 暂停对局

- 当前不做实际实现
- 只保留未来扩展空间

#### 继续对局

- 当前不做实际实现
- 只保留未来扩展空间

#### 单步推进

- 当前不做
- 适合后续更强调试模式再讨论

#### 终止对局

- 当前可以不做
- 如果后续需要反复快速测试，再评估加入

### 19.4 接口分层建议

建议第一阶段接口在逻辑上分成四组：

#### 会话接口

- 查询当前活动对局
- 创建对局
- 启动对局

#### 状态接口

- 获取当前完整状态

#### 事件接口

- 获取最近事件

#### 输入接口

- 提交输入

#### 复盘接口

- 查询复盘状态

### 19.5 错误处理原则

第一阶段建议所有接口都明确区分几类错误：

- 当前没有活动对局
- 当前已有活动对局，不允许重复创建
- 生命周期不允许当前操作
- 输入请求不存在或已过期
- 视角非法
- 后端运行异常

错误返回要尽量稳定，便于前端直接展示。

#### 统一返回外层结构

建议所有接口都使用统一外层结构，无论成功还是失败：

```json
{
  "ok": true,
  "data": {},
  "error": null,
  "meta": {
    "request_id": "req_srv_001",
    "timestamp": "2026-03-25T20:00:00+08:00"
  }
}
```

失败时：

```json
{
  "ok": false,
  "data": null,
  "error": {
    "code": "SESSION_NOT_FOUND",
    "message": "当前没有活动对局",
    "category": "session",
    "details": {},
    "retryable": false
  },
  "meta": {
    "request_id": "req_srv_002",
    "timestamp": "2026-03-25T20:00:01+08:00"
  }
}
```

推荐约束：

- `ok` 用于快速判断请求是否成功
- `data` 只在成功时返回业务数据
- `error` 只在失败时返回错误对象
- `meta` 保留服务端请求标识和时间戳，便于日志排查和前端调试

#### 错误对象建议字段

建议错误对象至少包含：

- `code`
- `message`
- `category`
- `details`
- `retryable`

字段含义：

- `code`
  - 稳定错误码，前端和日志都优先依赖它，而不是依赖中文文案
- `message`
  - 面向人类阅读的错误说明
- `category`
  - 用于快速归类错误来源
- `details`
  - 结构化补充信息，例如当前状态、请求参数、期望状态
- `retryable`
  - 表示调用方是否适合直接重试

#### 推荐错误分类

建议第一阶段先统一为以下几类：

- `session`
- `lifecycle`
- `input`
- `view`
- `review`
- `validation`
- `internal`

这样后续即使新增更细错误，也仍然可以落在既有分类下扩展。

#### 推荐错误码命名规则

建议错误码使用全大写下划线风格，并优先保持“稳定、可扩展、可读”：

- `SESSION_NOT_FOUND`
- `SESSION_ALREADY_EXISTS`
- `LIFECYCLE_INVALID_TRANSITION`
- `INPUT_REQUEST_NOT_FOUND`
- `INPUT_REQUEST_EXPIRED`
- `INPUT_SUBMIT_FORBIDDEN`
- `VIEW_TYPE_INVALID`
- `REVIEW_NOT_READY`
- `VALIDATION_FAILED`
- `INTERNAL_ERROR`

命名建议：

- 尽量先写领域，再写具体问题
- 新增错误码时，优先在已有分类下补充，而不是随意发散新前缀
- 不要把错误码和中文文案绑定

#### 推荐错误码与场景对应

第一阶段建议至少明确这些常见错误：

| 错误码 | 分类 | 典型场景 | `retryable` |
|------|------|------|------|
| `SESSION_NOT_FOUND` | `session` | 当前没有活动对局 | `false` |
| `SESSION_ALREADY_EXISTS` | `session` | 已有活动对局，重复创建 | `false` |
| `LIFECYCLE_INVALID_TRANSITION` | `lifecycle` | 当前状态不允许执行该操作 | `false` |
| `INPUT_REQUEST_NOT_FOUND` | `input` | 提交了不存在的 `request_id` | `false` |
| `INPUT_REQUEST_EXPIRED` | `input` | 输入请求已过期 | `false` |
| `INPUT_SUBMIT_FORBIDDEN` | `input` | 非当前目标玩家提交输入 | `false` |
| `VIEW_TYPE_INVALID` | `view` | 请求了不支持的视图 | `false` |
| `REVIEW_NOT_READY` | `review` | 复盘尚未生成完成 | `true` |
| `VALIDATION_FAILED` | `validation` | 请求字段缺失或格式错误 | `false` |
| `INTERNAL_ERROR` | `internal` | 未分类的服务端异常 | `true` |

#### 扩展性约束

为了避免后续实现时频繁改协议，建议现在就明确：

- 可以新增错误码，但不应轻易修改已发布错误码语义
- 可以在 `details` 中补充字段，但不应移除已有顶层字段
- 前端默认应对未知错误码做兜底展示，而不是假设错误码集合永远固定
- 同一类错误如果需要细分，应优先新增更具体的 `code`，而不是改变 `category` 结构

### 19.6 视角边界

第一阶段 API 默认以“上帝视角调试 UI”为主，因此必须接口优先服务上帝视角。

但设计时建议保留扩展位，避免后续玩家模式完全重做：

- 状态接口支持 `view_type`
- 事件接口支持可见性裁剪
- 输入接口支持 `player_id` 关联

### 19.7 第一阶段结论

如果只做第一阶段最小可用 UI，后端真正必须补出的能力可以压缩成：

1. 创建对局
2. 启动对局
3. 拉完整状态
4. 补拉事件
5. 提交输入
6. 查看复盘状态

这已经足够支撑：

- 单机上帝视角调试页
- 后续玩家模式的最小后端基础

## 20. 正式契约草案

本节不是最终 API 文档，但用于把前面已经确认的结构收敛成更接近实现的契约草案。

### 20.1 状态快照契约草案

#### 顶层结构

```json
{
  "game_id": "werewolf_1234",
  "session_id": "session_1234",
  "mode": "observer",
  "view_type": "god",
  "lifecycle_status": "running",
  "phase": "day_discussion",
  "day_number": 2,
  "night_number": 1,
  "president_id": 4,
  "alive_player_ids": [1, 2, 4, 5, 6],
  "players": [],
  "pending_input": null,
  "review_status": {},
  "result": null,
  "meta": {}
}
```

#### 字段要求

| 字段 | 必需 | 说明 |
|------|------|------|
| `game_id` | 是 | 游戏实例 ID |
| `session_id` | 是 | 运行层会话 ID |
| `mode` | 是 | `observer` 或 `player` |
| `view_type` | 是 | `god` 或 `player` |
| `lifecycle_status` | 是 | 会话生命周期状态 |
| `phase` | 是 | 当前阶段 |
| `day_number` | 是 | 当前天数 |
| `night_number` | 是 | 当前夜数 |
| `president_id` | 否 | 当前警长 ID，没有则为 `null` |
| `alive_player_ids` | 是 | 当前存活玩家 ID 列表 |
| `players` | 是 | 视角化玩家状态列表 |
| `pending_input` | 否 | 没有待输入时为 `null` |
| `review_status` | 是 | 复盘状态对象 |
| `result` | 否 | 游戏未结束时为 `null` |
| `meta` | 否 | 扩展元数据 |

#### 玩家状态结构

```json
{
  "player_id": 1,
  "display_name": "1号玩家",
  "is_alive": true,
  "role": "werewolf",
  "public_role": null,
  "camp": "werewolf",
  "is_human": false,
  "personality": "逻辑严谨型",
  "death_cause": null,
  "has_last_words": false,
  "is_president": false
}
```

字段约束：

- `role` 只在上帝视角或本人视角返回真实值
- `camp` 默认不对普通玩家视角开放
- `public_role` 用于表达公开身份，而不是替代真实身份

#### `pending_input` 结构

```json
{
  "request_id": "req_001",
  "input_type": "vote",
  "player_id": 3,
  "prompt": "请选择投票对象",
  "options": [1, 2, 4],
  "timeout": 60,
  "metadata": {}
}
```

#### `review_status` 结构

```json
{
  "status": "running",
  "summary": null,
  "paths": null,
  "error": null
}
```

### 20.5 第一阶段 API 契约草案
